  • Viết producer/consumer tránh mất message, duplicate vô tội vạ và lag khó kiểm soát.
  • Biết dùng Kafka Connect cho tích hợp hệ thống và Python stream processor cho xử lý event liên tục.
  • Vận hành được cluster KRaft cơ bản: metrics, alert, security, capacity và upgrade.

Stage 1: Foundation - chạy Kafka local

Mental model cần nhớ

  • Event: bản ghi gồm key, value, timestamp và headers.
  • Topic: nơi lưu event theo dòng thời gian, có retention thay vì xoá ngay sau khi đọc.
  • Partition: đơn vị song song và ordering. Cùng key sẽ đi vào cùng partition nếu partition count không đổi.
  • Consumer group: nhiều consumer chia nhau đọc partitions. Một partition chỉ do một consumer trong cùng group xử lý tại một thời điểm.
  • Offset: vị trí đọc của consumer. Commit offset là quyết định nghiệp vụ, không chỉ là kỹ thuật.

Quickstart bằng Docker

docker pull apache/kafka:4.3.1
docker run --name kafka-dev -p 9092:9092 apache/kafka:4.3.1

# Terminal khác
docker exec -it kafka-dev /opt/kafka/bin/kafka-topics.sh \
  --create --topic invoice-events \
  --partitions 3 --replication-factor 1 \
  --bootstrap-server localhost:9092

docker exec -it kafka-dev /opt/kafka/bin/kafka-console-producer.sh \
  --topic invoice-events --bootstrap-server localhost:9092

docker exec -it kafka-dev /opt/kafka/bin/kafka-console-consumer.sh \
  --topic invoice-events --from-beginning \
  --bootstrap-server localhost:9092

Dùng `--bootstrap-server` cho Kafka 4.x. Không thiết kế bài học mới quanh ZooKeeper vì KRaft là hướng mặc định của Kafka hiện đại.

Quyết định Gợi ý bắt đầu Khi nào đổi?
Topic name domain.event.v1, ví dụ invoice.issued.v1 Khi schema hoặc semantic thay đổi không tương thích.

Stage 2: Producer & Consumer correctness

Producer rules

  • Dùng key ổn định cho event cần giữ thứ tự theo entity.
  • Đặt acks=all và enable.idempotence=true cho event quan trọng.
  • Không retry vô hạn nếu downstream tạo side effect không idempotent.
  • Đo latency theo percentiles, không chỉ nhìn average.
python3 -m venv .venv
source .venv/bin/activate
pip install "confluent-kafka==2.15.0"
import json
import socket
from confluent_kafka import Producer

delivery_errors = []

def on_delivery(err, msg):
    if err is not None:
        delivery_errors.append(err)
        return
    print(f"delivered topic={msg.topic()} partition={msg.partition()} offset={msg.offset()}")

producer = Producer({
    "bootstrap.servers": "localhost:9092",
    "client.id": f"invoice-producer-{socket.gethostname()}",
    "acks": "all",
    "enable.idempotence": True,
    "delivery.timeout.ms": 120000,
})

event = {
    "invoice_id": "INV-001",
    "tax_code": "0101243150",
    "amount": 1200000,
    "currency": "VND",
}

producer.produce(
    "invoice.issued.v1",
    key=event["tax_code"],
    value=json.dumps(event, ensure_ascii=False).encode("utf-8"),
    callback=on_delivery,
)
producer.flush(10)

if delivery_errors:
    raise RuntimeError(delivery_errors[0])

Consumer rules

  • Commit offset sau khi xử lý thành công nếu cần at-least-once.
  • Giữ processing idempotent vì at-least-once có thể đọc lại message.
  • Dùng dead-letter topic cho lỗi dữ liệu không thể retry.
  • Đặt giới hạn xử lý theo batch để tránh vượt max.poll.interval.ms.
import json
from confluent_kafka import Consumer, KafkaError, KafkaException

def validate_invoice(event):
    required = ["invoice_id", "tax_code", "amount"]
    missing = [field for field in required if field not in event]
    if missing:
        raise ValueError(f"missing fields: {missing}")
    if event["amount"] <= 0:
        raise ValueError("amount must be positive")

consumer = Consumer({
    "bootstrap.servers": "localhost:9092",
    "group.id": "invoice-validator",
    "enable.auto.commit": False,
    "auto.offset.reset": "earliest",
})

consumer.subscribe(["invoice.issued.v1"])

try:
    while True:
        msg = consumer.poll(1.0)
        if msg is None:
            continue
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                continue
            raise KafkaException(msg.error())

        event = json.loads(msg.value().decode("utf-8"))
        validate_invoice(event)
        consumer.commit(message=msg, asynchronous=False)
finally:
    consumer.close()

Delivery semantics

Mode Ý nghĩa Dùng khi
At-most-once Có thể mất message, ít duplicate. Telemetry không quan trọng từng event.
At-least-once Không mất nếu retry đúng, có thể duplicate. Phần lớn nghiệp vụ backend.
Exactly-once Producer transaction + consumer isolation + sink hỗ trợ. Stream processing có state hoặc write lại Kafka.

Stage 3: KRaft architecture

KRaft core

KRaft thay ZooKeeper bằng metadata quorum nằm trong Kafka. Controller quản lý metadata, broker phục vụ data plane. Khi vận hành production, nên tách vai trò controller và broker để giảm blast radius.

  • Controller quorum: thường 3 node để chịu lỗi 1 node.
  • Broker: nơi chứa partition log và phục vụ producer/consumer.
  • Metadata log: nguồn sự thật cho topic, partition, ACL, configs.
  • ISR: tập replica đang bắt kịp leader, ảnh hưởng trực tiếp tới durability.

Compatibility notes

  • Kafka 4.x thay đổi mạnh ở phía broker/runtime; khi chạy bằng Docker image chính thức bạn không cần cài runtime riêng trên máy học.
  • confluent-kafka là Python binding trên librdkafka; vẫn cần regression test producer/consumer trước khi upgrade broker.
  • Các lệnh admin cũ dùng --zookeeper đã bị loại bỏ. Dùng --bootstrap-server.
  • Cluster ZooKeeper cũ cần kế hoạch migration riêng, không nâng bừa trong cùng một maintenance window ngắn.

Topology production nhỏ

  kafka-ctrl-1
  kafka-ctrl-2
  kafka-ctrl-3

Stage 4: Kafka Connect & integration

Khi nào dùng Kafka Connect?

Dùng Connect khi bạn cần đưa dữ liệu vào/ra Kafka theo cách lặp lại được: database CDC, file/object storage, search index, warehouse, audit stream hoặc bridge giữa hệ thống batch và streaming.

  • Standalone mode: local dev, demo, job nhỏ.
  • Distributed mode: production, nhiều worker, REST API quản lý connector.
  • Offset management: Connect quản lý offset để connector developer không tự xử lý phần dễ lỗi này.

Design pattern cho e-invoice

Invoice service
  -> outbox table
  -> source connector / CDC
  -> topic invoice.issued.v1
  -> validator consumer
  -> topic invoice.validated.v1
  -> sink connector to warehouse/object storage

Connect checklist

  • Mỗi connector có owner, SLA và dashboard riêng.
  • Không deploy plugin lạ vào cluster production trước khi scan license và security.
  • Đặt retry/backoff rõ ràng; lỗi dữ liệu nên đi DLT, lỗi hạ tầng nên retry.
  • Theo dõi connector task failed, source lag, sink latency và offset commit error.

Stage 5: Python stream processing

Python stream processor mental model

Bản Python của khoá học dùng pattern consume → transform → produce. State nhẹ có thể giữ trong process để học, còn production lớn nên cân nhắc Flink, Spark Structured Streaming hoặc một state store ngoài. 

import json
from collections import defaultdict
from confluent_kafka import Consumer, Producer, KafkaException

WINDOW_SECONDS = 300
state = defaultdict(lambda: {"count": 0, "amount": 0})

consumer = Consumer({
    "bootstrap.servers": "localhost:9092",
    "group.id": "invoice-summary-worker",
    "enable.auto.commit": False,
    "auto.offset.reset": "earliest",
})
producer = Producer({"bootstrap.servers": "localhost:9092", "enable.idempotence": True})
consumer.subscribe(["invoice.issued.v1"])

def window_start(epoch_seconds):
    return int(epoch_seconds // WINDOW_SECONDS) * WINDOW_SECONDS

try:
    while True:
        msg = consumer.poll(1.0)
        if msg is None:
            continue
        if msg.error():
            raise KafkaException(msg.error())

        event = json.loads(msg.value().decode("utf-8"))
        bucket = window_start(event.get("issued_at_epoch", 0))
        key = (event["tax_code"], bucket)
        state[key]["count"] += 1
        state[key]["amount"] += event["amount"]

        summary = {
            "tax_code": event["tax_code"],
            "window_start": bucket,
            "count": state[key]["count"],
            "amount": state[key]["amount"],
        }
        producer.produce(
            "invoice.summary.5m.v1",
            key=event["tax_code"],
            value=json.dumps(summary).encode("utf-8"),
        )
        producer.poll(0)
        consumer.commit(message=msg, asynchronous=False)
finally:
    producer.flush(10)
    consumer.close()

Quyết định khi dùng Streams

  • Chọn window theo nghiệp vụ, không theo cảm giác. Invoice daily report khác fraud detection realtime.
  • State trong memory chỉ dùng cho bài học hoặc job nhỏ; production cần checkpoint hoặc external state store.
  • Output topic nên có key ổn định để downstream deduplicate dễ hơn.
  • Với exception, phân loại: lỗi dữ liệu, lỗi code, lỗi downstream và lỗi hạ tầng.

Stage 6: Production operations

Monitoring tối thiểu

Nhóm metric Cần nhìn Ý nghĩa
Broker health Under replicated partitions, offline partitions, request latency Cluster có còn phục vụ ổn không.
Consumer Consumer lag, rebalance rate, poll latency Consumer có theo kịp producer không.
Storage Disk usage, log segment growth, retention pressure Nguy cơ đầy disk và mất khả năng ghi.
Controller Active controller, metadata log latency, quorum health KRaft metadata plane có ổn định không.

Security baseline

  • Bật TLS cho traffic ngoài dev laptop.
  • Dùng SASL/OAuth hoặc mTLS theo chuẩn hạ tầng hiện có.
  • ACL theo service account, không share user giữa nhiều app.
  • Không log raw payload nếu có PII, invoice data hoặc secret.

New consumer rebalance protocol

Kafka 4.x có consumer rebalance protocol thế hệ mới. Server hỗ trợ từ 4.0, consumer cần bật group.protocol=consumer. Nên thử ở staging trước vì thay đổi này ảnh hưởng cách group assign partition. 

group.id=invoice-validator
group.protocol=consumer
group.remote.assignor=uniform
enable.auto.commit=false
max.poll.records=500

Anti-patterns cần tránh

  • Dùng Kafka như database chính: Kafka lưu event log tốt, nhưng query linh hoạt vẫn cần storage/query engine phù hợp.
  • Partition quá nhiều ngay từ đầu: tăng overhead controller, file handle và recovery time.
  • Message quá lớn: đưa blob/XML lớn vào object storage, chỉ gửi reference + metadata qua Kafka.
  • Consumer side effect không idempotent: retry sẽ tạo duplicate order, duplicate invoice hoặc gọi API nhiều lần.
  • Không có DLT: một record xấu có thể chặn cả partition.
  • Không đo lag theo consumer group: chỉ nhìn broker healthy vẫn chưa biết pipeline có đang trễ không.

Troubleshooting nhanh

Triệu chứng Nguyên nhân thường gặp Hướng xử lý
Consumer lag tăng đều Consumer chậm, partition ít, downstream nghẽn Tối ưu xử lý, tăng partition/consumer, kiểm tra DB/API downstream.
Rebalance liên tục Consumer poll quá lâu hoặc session timeout thấp Giảm batch, tăng timeout hợp lý, kiểm tra GC và network.
Duplicate message At-least-once + retry/commit lại Làm sink idempotent, dùng key tự nhiên, ghi dedup table nếu cần.
Producer timeout Broker overload, ISR thiếu, request queue đầy Kiểm tra broker latency, min ISR, disk/network và throttling.
Disk đầy nhanh Retention quá dài, message lớn, topic nội bộ tăng Rà retention, compaction, tiered storage và archive policy.
