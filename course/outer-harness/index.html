Outer Harness Learning Path

Outer harness là lớp phần mềm đứng ngoài model: nó quyết định input nào đi vào model, tool nào được gọi, state nào được lưu, lỗi nào được retry, hành động nào cần chặn, và kết quả nào đủ tốt để trả về. Với agent production, chất lượng không chỉ đến từ model hay prompt, mà đến từ cấu hình model + harness + dữ liệu + đánh giá.

Cập nhật nghiên cứu: 09/07/2026. Khoá học tổng hợp từ hướng nghiên cứu harness engineering, tài liệu LangChain/LangGraph, OpenAI Agents SDK, Pydantic AI và kinh nghiệm thiết kế backend Python cho hệ thống có tool execution.

Outer harness là gì?

Trong khóa học này, harness là mọi thứ bao quanh model call: prompt, message history, tool registry, memory, retry policy, budget, sandbox, permissions, observability và evaluator. Outer harness là lớp điều phối bên ngoài agent loop đơn lẻ: nó chạy nhiều candidate, đọc trace, phân loại failure, thay đổi prompt/tool/context/policy, rồi đo lại bằng eval.

Khái niệm Vai trò Ví dụ trong Python

Prompt Instruction và policy mềm cho model. system_prompt + task + context

Stage 1: Mental model

Agent thường fail ở đâu?

  • Context sai: model thiếu thông tin cần thiết hoặc nhận quá nhiều nhiễu.
  • Tool sai: tool schema mơ hồ, output dài, permission quá rộng hoặc side effect không kiểm soát.
  • State sai: agent quên quyết định trước đó, ghi nhớ dữ liệu nhạy cảm, hoặc replay không được.
  • Loop sai: retry vô hạn, gọi tool lặp, kết thúc sớm, hoặc không phân biệt lỗi tạm thời với lỗi nghiệp vụ.
  • Eval sai: chỉ nhìn final answer, không kiểm trace, chi phí, latency, tool call và evidence.

Trace-first thinking

Khi thiết kế harness, hãy coi mỗi lần chạy là một bản ghi audit. Trace tối thiểu cần có: request id, user intent, prompt version, model version, tool calls, tool results, token usage, guardrail decisions, final output, lỗi và evaluator score.

Một agent khó debug là agent không có trace. Một harness tốt luôn giúp bạn trả lời được: model đã thấy gì, đã làm gì, vì sao được phép làm vậy, và kết quả được kiểm bằng gì.

Stage 3: Core schemas

Message, tool call và state

# app/models.py
from __future__ import annotations

from enum import Enum
from typing import Any, Literal
from pydantic import BaseModel, Field


Role = Literal["system", "user", "assistant", "tool"]


class Message(BaseModel):
    role: Role
    content: str
    name: str | None = None


class ToolCall(BaseModel):
    name: str
    arguments: dict[str, Any] = Field(default_factory=dict)
    call_id: str


class ToolResult(BaseModel):
    call_id: str
    name: str
    ok: bool
    content: str
    metadata: dict[str, Any] = Field(default_factory=dict)


class FinishReason(str, Enum):
    FINAL = "final"
    TOOL_CALL = "tool_call"
    REFUSAL = "refusal"
    ERROR = "error"


class ModelResponse(BaseModel):
    content: str = ""
    finish_reason: FinishReason
    tool_calls: list[ToolCall] = Field(default_factory=list)
    usage: dict[str, int] = Field(default_factory=dict)


class AgentState(BaseModel):
    run_id: str
    messages: list[Message] = Field(default_factory=list)
    tool_results: list[ToolResult] = Field(default_factory=list)
    turn: int = 0
    cost_usd: float = 0.0
    metadata: dict[str, Any] = Field(default_factory=dict)

Config và budget

# app/models.py
class HarnessConfig(BaseModel):
    prompt_version: str = "invoice-copilot@0.1.0"
    max_turns: int = 8
    max_tool_calls: int = 12
    max_context_chars: int = 24_000
    tool_timeout_seconds: float = 8.0
    require_approval_for_writes: bool = True


class RunBudget(BaseModel):
    turns_left: int
    tool_calls_left: int

    def consume_turn(self) -> None:
        if self.turns_left <= 0:
            raise RuntimeError("Turn budget exceeded")
        self.turns_left -= 1

    def consume_tool_call(self) -> None:
        if self.tool_calls_left <= 0:
            raise RuntimeError("Tool budget exceeded")
        self.tool_calls_left -= 1

Schema không chỉ để validation. Nó là contract giữa model adapter, tool dispatcher, trace writer và evaluator. Khi contract rõ, bạn có thể replay, test và thay model mà không đổi toàn bộ hệ thống.

Stage 5: Tool registry

Tool là API nội bộ có contract

# app/tools.py
from __future__ import annotations

import inspect
from collections.abc import Awaitable, Callable
from typing import Any
from pydantic import BaseModel, Field, ValidationError


class ToolSpec(BaseModel):
    name: str
    description: str
    input_model: type[BaseModel]
    handler: Callable[[BaseModel], Awaitable[str]]
    side_effect: bool = False
    requires_approval: bool = False

    def to_json_schema(self) -> dict[str, Any]:
        return {
            "type": "function",
            "function": {
                "name": self.name,
                "description": self.description,
                "parameters": self.input_model.model_json_schema(),
            },
        }


class ToolRegistry:
    def __init__(self) -> None:
        self._tools: dict[str, ToolSpec] = {}

    def register(self, spec: ToolSpec) -> None:
        if spec.name in self._tools:
            raise ValueError(f"Duplicate tool: {spec.name}")
        if not inspect.iscoroutinefunction(spec.handler):
            raise TypeError("Tool handler must be async")
        self._tools[spec.name] = spec

    def schemas(self) -> list[dict[str, Any]]:
        return [tool.to_json_schema() for tool in self._tools.values()]

    async def call(self, name: str, arguments: dict[str, Any]) -> str:
        spec = self._tools.get(name)
        if spec is None:
            raise KeyError(f"Unknown tool: {name}")
        try:
            parsed = spec.input_model.model_validate(arguments)
        except ValidationError as exc:
            raise ValueError(f"Invalid arguments for {name}: {exc}") from exc
        return await spec.handler(parsed)

Ví dụ tool tra cứu invoice

# app/tools.py
class GetInvoiceArgs(BaseModel):
    invoice_id: str = Field(min_length=3, max_length=64)


async def get_invoice(args: GetInvoiceArgs) -> str:
    fake_db = {
        "INV-001": {
            "status": "issued",
            "amount": 1_250_000,
            "tax_code": "0312345678",
        }
    }
    invoice = fake_db.get(args.invoice_id)
    if invoice is None:
        return "Invoice not found"
    return (
        f"invoice_id={args.invoice_id}; "
        f"status={invoice['status']}; "
        f"amount={invoice['amount']}; "
        f"tax_code={invoice['tax_code']}"
    )


def build_registry() -> ToolRegistry:
    registry = ToolRegistry()
    registry.register(
        ToolSpec(
            name="get_invoice",
            description="Lookup invoice status, amount and tax code by invoice id.",
            input_model=GetInvoiceArgs,
            handler=get_invoice,
        )
    )
    return registry

Quy tắc thiết kế tool

  • Tool name dùng động từ rõ: get_invoice, search_policy, create_refund_draft.
  • Input schema nhỏ, kiểu dữ liệu rõ, không nhận blob tự do nếu không cần.
  • Output ngắn, có cấu trúc, không trả nguyên tài liệu dài vào context.
  • Tool có side effect phải idempotent hoặc có approval gate.
  • Không để model tự quyết quyền truy cập. Harness phải kiểm permission trước khi dispatch.

Stage 6: Agent loop

Loop tối thiểu

# app/harness.py
from app.models import (
    AgentState,
    FinishReason,
    HarnessConfig,
    Message,
    RunBudget,
    ToolResult,
)
from app.tools import ToolRegistry


class AgentHarness:
    def __init__(
        self,
        model: ModelClient,
        tools: ToolRegistry,
        config: HarnessConfig | None = None,
    ) -> None:
        self.model = model
        self.tools = tools
        self.config = config or HarnessConfig()

    async def run(self, run_id: str, user_input: str) -> AgentState:
        state = AgentState(run_id=run_id)
        state.messages.extend([
            Message(role="system", content=self._system_prompt()),
            Message(role="user", content=user_input),
        ])
        budget = RunBudget(
            turns_left=self.config.max_turns,
            tool_calls_left=self.config.max_tool_calls,
        )

        while True:
            budget.consume_turn()
            state.turn += 1
            response = await self.model.complete(
                messages=self._trim_context(state.messages),
                tools=self.tools.schemas(),
            )

            if response.finish_reason == FinishReason.FINAL:
                state.messages.append(Message(role="assistant", content=response.content))
                return state

            if response.finish_reason == FinishReason.TOOL_CALL:
                await self._handle_tool_calls(state, response.tool_calls, budget)
                continue

            state.messages.append(
                Message(role="assistant", content=f"Run stopped: {response.finish_reason}")
            )
            return state

    async def _handle_tool_calls(self, state, tool_calls, budget) -> None:
        for call in tool_calls:
            budget.consume_tool_call()
            try:
                content = await self.tools.call(call.name, call.arguments)
                result = ToolResult(
                    call_id=call.call_id,
                    name=call.name,
                    ok=True,
                    content=content,
                )
            except Exception as exc:
                result = ToolResult(
                    call_id=call.call_id,
                    name=call.name,
                    ok=False,
                    content=str(exc),
                )

            state.tool_results.append(result)
            state.messages.append(
                Message(
                    role="tool",
                    name=result.name,
                    content=result.content,
                )
            )

    def _system_prompt(self) -> str:
        return (
            "You are an invoice operations assistant. "
            "Use tools when facts are needed. "
            "Never invent invoice status or amounts. "
            "If evidence is insufficient, say what is missing."
        )

    def _trim_context(self, messages: list[Message]) -> list[Message]:
        joined = []
        total = 0
        for msg in reversed(messages):
            total += len(msg.content)
            if total > self.config.max_context_chars:
                break
            joined.append(msg)
        return list(reversed(joined))

Invariants cần giữ

  • Mỗi tool call phải tạo đúng một tool result, kể cả khi lỗi.
  • Mỗi run có giới hạn turn và tool call.
  • Final answer chỉ được trả sau khi guardrail output pass.
  • Tool side effect không chạy nếu approval chưa có.
  • Trace ghi cả success lẫn failure, không chỉ ghi final output.

Stage 7: Context và memory

Context assembler

Context assembler là thành phần quan trọng nhất sau tool loop. Nó chọn dữ liệu nào được đưa vào model ở từng turn. Cùng một model nhưng context assembler khác nhau có thể tạo ra chất lượng agent rất khác nhau. 

# app/memory.py
from pydantic import BaseModel


class MemoryItem(BaseModel):
    key: str
    value: str
    scope: str = "user"
    importance: int = 1


class InMemoryStore:
    def __init__(self) -> None:
        self._items: dict[str, MemoryItem] = {}

    def put(self, item: MemoryItem) -> None:
        self._items[item.key] = item

    def search(self, query: str, limit: int = 5) -> list[MemoryItem]:
        terms = set(query.lower().split())
        scored = []
        for item in self._items.values():
            text = f"{item.key} {item.value}".lower()
            score = sum(1 for term in terms if term in text) + item.importance
            scored.append((score, item))
        scored.sort(key=lambda pair: pair[0], reverse=True)
        return [item for score, item in scored[:limit] if score > 0]

  • System prompt và policy: luôn giữ.
  • Current user request: luôn giữ.
  • Tool results mới nhất: giữ nếu trực tiếp liên quan.
  • Evidence retrieved: giới hạn top-k và cắt theo section.
  • History cũ: summarize hoặc bỏ nếu không liên quan.
  • Debug trace: không đưa vào model trừ khi đang chạy repair flow.

Stage 8: Guardrails và recovery

Input guardrail

# app/guardrails.py
from pydantic import BaseModel


class GuardrailDecision(BaseModel):
    allowed: bool
    reason: str = ""
    redacted_input: str | None = None


class InputGuardrail:
    blocked_terms = {"api_key", "private_key", "password="}

    def check(self, user_input: str) -> GuardrailDecision:
        lowered = user_input.lower()
        for term in self.blocked_terms:
            if term in lowered:
                return GuardrailDecision(
                    allowed=False,
                    reason=f"Sensitive token-like content detected: {term}",
                )
        return GuardrailDecision(allowed=True)

Output contract bằng Pydantic

# app/models.py
class Evidence(BaseModel):
    source: str
    quote_or_fact: str


class FinalAnswer(BaseModel):
    answer: str
    confidence: float = Field(ge=0.0, le=1.0)
    evidence: list[Evidence] = Field(default_factory=list)
    missing_information: list[str] = Field(default_factory=list)


def validate_final_answer(data: dict) -> FinalAnswer:
    answer = FinalAnswer.model_validate(data)
    if answer.confidence >= 0.7 and not answer.evidence:
        raise ValueError("High-confidence answer must include evidence")
    return answer

Recovery policy

Lỗi Phân loại Harness xử lý
Model timeout Transient Retry exponential backoff, đổi model fallback nếu hết retry.
Tool validation error Model/tool contract Đưa lỗi ngắn về model một lần, sau đó stop nếu lặp lại.
Permission denied Policy Không retry, yêu cầu approval hoặc trả lý do bị chặn.
Evidence missing Grounding Yêu cầu retrieval/tool call bổ sung hoặc trả missing info.
Eval regression Release gate Không deploy config mới, mở failure report.

Stage 9: Tracing

Trace writer tối thiểu

# app/tracing.py
from __future__ import annotations

import json
import time
from pathlib import Path
from typing import Any


class JsonlTraceWriter:
    def __init__(self, path: str = "traces/runs.jsonl") -> None:
        self.path = Path(path)
        self.path.parent.mkdir(parents=True, exist_ok=True)

    def event(self, run_id: str, event_type: str, payload: dict[str, Any]) -> None:
        row = {
            "ts": time.time(),
            "run_id": run_id,
            "event_type": event_type,
            "payload": payload,
        }
        with self.path.open("a", encoding="utf-8") as f:
            f.write(json.dumps(row, ensure_ascii=False) + "\n")

Các event nên ghi

  • run_started: request, user id hash, prompt version, config.
  • model_started và model_finished: latency, usage, finish reason.
  • tool_started và tool_finished: tool name, arguments redacted, status.
  • guardrail_decision: policy, allowed, reason.
  • eval_score: metric, score, explanation.
  • run_finished: final status, cost, latency, error class.

Trace JSONL là đủ cho local. Production nên dùng OpenTelemetry hoặc nền tảng tracing/eval chuyên dụng để xem span, latency, token usage và replay theo run id.

# tests/test_evals.py
import json
from pathlib import Path

import pytest


def load_cases():
    path = Path("eval_datasets/invoice_cases.jsonl")
    return [json.loads(line) for line in path.read_text().splitlines() if line.strip()]


@pytest.mark.asyncio
@pytest.mark.parametrize("case", load_cases())
async def test_invoice_eval(case, harness_factory):
    harness = harness_factory(case)
    state = await harness.run(run_id=case["id"], user_input=case["input"])
    transcript = "\n".join(message.content for message in state.messages)

    for fact in case.get("expected_facts", []):
        assert fact in transcript

    called_tools = [result.name for result in state.tool_results]
    for tool_name in case.get("must_call_tools", []):
        assert tool_name in called_tools

Metric nên đo

  • Task success: agent có giải đúng việc người dùng cần không?
  • Groundedness: kết luận có dựa vào tool/evidence không?
  • Tool correctness: gọi đúng tool, đúng args, đúng thứ tự không?
  • Policy safety: có chặn hành động rủi ro và dữ liệu nhạy cảm không?
  • Efficiency: số turn, số tool call, token, latency và cost.
  • Recoverability: khi tool lỗi, agent có dừng/khôi phục đúng không?

Stage 11: RAG harness

Pipeline RAG trong harness

  1. Normalize câu hỏi và trích intent.
  2. Chọn retriever phù hợp: keyword, vector, SQL hoặc hybrid.
  3. Retrieve top-k nhỏ, lọc quyền truy cập theo user/context.
  4. Rerank hoặc dedupe chunks.
  5. Cắt context theo budget, giữ metadata nguồn.
  6. Ép final answer chứa evidence hoặc missing information.
# app/prompts.py
def build_rag_context(chunks: list[dict], max_chars: int = 8_000) -> str:
    parts: list[str] = []
    total = 0
    for idx, chunk in enumerate(chunks, start=1):
        text = chunk["text"].strip()
        source = chunk.get("source", "unknown")
        block = f"[{idx}] source={source}\n{text}\n"
        if total + len(block) > max_chars:
            break
        parts.append(block)
        total += len(block)
    return "\n".join(parts)

RAG anti-patterns

  • Đưa top-20 chunks vào prompt vì sợ thiếu context.
  • Không lưu source id nên final answer không audit được.
  • Retriever bỏ qua permission của user.
  • Không có eval riêng cho retrieval, chỉ eval final answer.
  • Không phân biệt "không tìm thấy" với "model không biết".

Stage 12: Multi-agent orchestration

Khi nào cần nhiều agent?

Multi-agent hữu ích khi task có nhiều vai trò tách biệt: researcher, planner, executor, reviewer, hoặc khi cần chạy song song trong context riêng. Không nên dùng multi-agent chỉ để làm hệ thống trông phức tạp hơn. Mỗi agent thêm vào là thêm chi phí, latency và failure mode.

Pattern Dùng khi Rủi ro
Manager-worker Main agent chia việc cho subagents. Manager tổng hợp sai hoặc bỏ sót evidence.
Planner-executor Cần kế hoạch ổn định trước khi hành động. Plan đẹp nhưng executor không kiểm tool feedback.
Actor-reviewer Cần review độc lập trước khi trả output. Reviewer cũng có thể hallucinate nếu thiếu evidence.
Specialist handoff Routing sang domain chuyên biệt. Handoff sai intent hoặc mất context quan trọng.

Orchestrator đơn giản

# app/harness.py
class Orchestrator:
    def __init__(self, planner: AgentHarness, executor: AgentHarness, reviewer: AgentHarness):
        self.planner = planner
        self.executor = executor
        self.reviewer = reviewer

    async def run(self, run_id: str, user_input: str):
        plan_state = await self.planner.run(f"{run_id}:plan", user_input)
        plan = plan_state.messages[-1].content

        exec_state = await self.executor.run(
            f"{run_id}:exec",
            f"Execute this plan with tools:\n{plan}",
        )
        draft = exec_state.messages[-1].content

        review_state = await self.reviewer.run(
            f"{run_id}:review",
            f"Review this answer for evidence and policy issues:\n{draft}",
        )
        return {
            "plan": plan_state,
            "execution": exec_state,
            "review": review_state,
        }

Stage 13: Framework mapping

Tự xây hay dùng framework?

Lựa chọn Phù hợp khi Ghi chú
Tự xây Python loop Flow hẹp, cần kiểm soát contract, muốn test vendor-neutral. Tốt để học và cho production nhỏ có yêu cầu rõ.
OpenAI Agents SDK Dùng OpenAI stack, cần tool loop, guardrails, sessions, tracing. Python-first, ít primitive, có runtime agent sẵn.
LangChain agent Cần nhiều integration, middleware, tool abstraction. Agent = model gọi tool trong loop, harness cấu hình qua middleware.
LangGraph Workflow dài, có state graph, persistence, human-in-the-loop. Phù hợp khi agent cần resume, checkpoint và state rõ.
Pydantic AI Muốn agent schema-first, dependency injection và typed output. Hợp với team Python đã dùng Pydantic/FastAPI.

Port sang LangGraph ở mức ý tưởng

# LangGraph sketch, không thay thế lõi tự xây ở các stage trước
from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict


class InvoiceState(TypedDict):
    messages: list[dict]
    invoice_id: str | None
    evidence: list[str]
    final_answer: str | None


def route_intent(state: InvoiceState):
    return {"invoice_id": "INV-001"}


def lookup_invoice(state: InvoiceState):
    return {"evidence": ["status=issued", "amount=1250000"]}


def answer(state: InvoiceState):
    return {"final_answer": "INV-001 đã phát hành, số tiền 1.250.000 VND."}


builder = StateGraph(InvoiceState)
builder.add_node("route_intent", route_intent)
builder.add_node("lookup_invoice", lookup_invoice)
builder.add_node("answer", answer)
builder.add_edge(START, "route_intent")
builder.add_edge("route_intent", "lookup_invoice")
builder.add_edge("lookup_invoice", "answer")
builder.add_edge("answer", END)
graph = builder.compile()

Mapping quan trọng: AgentState tự xây tương ứng với graph state, tool dispatcher tương ứng với node/tool node, trace writer tương ứng với observability, checkpointer tương ứng với persistence.

Production checklist

  • Auth và tenant boundary rõ ràng trước mọi retrieval/tool call.
  • Rate limit theo user, tenant và model provider.
  • Timeout riêng cho model, tool, toàn bộ run và streaming connection.
  • Secret không đi vào prompt, trace hoặc eval dataset.
  • Redaction cho tool arguments và tool results nhạy cảm.
  • Prompt/tool/config version được lưu trong trace.
  • Eval suite chạy trong CI trước khi đổi prompt, model hoặc tool schema.
  • Human approval cho action ghi dữ liệu, gửi email, thanh toán, xóa file hoặc gọi API chi phí cao.

Capstone: Invoice Operations Copilot

Phạm vi dự án

Dự án cuối khóa là một copilot cho nghiệp vụ hóa đơn. Người dùng hỏi trạng thái hóa đơn, đối chiếu thông tin, tìm chính sách xử lý lỗi, tạo draft hành động và nhận câu trả lời có evidence. Agent không được tự ý phát hành/hủy/sửa hóa đơn nếu thiếu approval.

Yêu cầu chức năng

  • Tool get_invoice(invoice_id) trả status, amount, tax code.
  • Tool search_policy(query) trả policy chunks có source id.
  • Tool create_action_draft(invoice_id, action) yêu cầu approval.
  • Final answer có schema FinalAnswer với confidence và evidence.
  • Trace JSONL ghi mọi model step, tool call, guardrail decision và eval score.
  • Eval dataset tối thiểu 30 case: lookup, missing invoice, policy question, prohibited action, tool failure.

Rubric chấm điểm

Tiêu chí Đạt Tốt
Correctness Trả đúng fact ở 80% eval case. Trả đúng fact ở 95% eval case, không bịa khi thiếu dữ liệu.
Tool use Gọi đúng tool cho lookup cơ bản. Gọi tool tối thiểu cần thiết, args valid, xử lý tool error tốt.
Safety Chặn action ghi dữ liệu thiếu approval. Redact dữ liệu nhạy cảm, trace không lộ secret, policy gate rõ.
Observability Có trace JSONL theo run id. Có span/event đầy đủ, replay được failure và so sánh config.
Production shape Có FastAPI endpoint chạy được. Có timeout, rate limit hook, health check, config version và CI eval.

Outer loop improvement

Sau khi có harness chạy được, outer loop là cách cải thiện có hệ thống. Mỗi vòng gồm: chạy eval, đọc trace, phân loại lỗi, tạo candidate config, chạy lại cùng dataset, so sánh metric rồi chỉ promote khi thắng trên cả chất lượng lẫn chi phí. 

# app/evals.py
from dataclasses import dataclass


@dataclass(frozen=True)
class CandidateConfig:
    name: str
    prompt_version: str
    max_turns: int
    retriever_top_k: int


async def compare_candidates(cases, candidates, harness_factory):
    report = []
    for candidate in candidates:
        passed = 0
        total_tool_calls = 0
        for case in cases:
            harness = harness_factory(candidate)
            state = await harness.run(case["id"], case["input"])
            transcript = "\n".join(message.content for message in state.messages)
            ok = all(fact in transcript for fact in case.get("expected_facts", []))
            passed += int(ok)
            total_tool_calls += len(state.tool_results)
        report.append({
            "candidate": candidate.name,
            "accuracy": passed / len(cases),
            "avg_tool_calls": total_tool_calls / len(cases),
        })
    return sorted(report, key=lambda row: row["accuracy"], reverse=True)

Failure taxonomy

  • retrieval_miss: evidence đúng không được retrieve.
  • context_noise: evidence đúng có trong context nhưng bị nhiễu che lấp.
  • tool_arg_error: model gọi tool với args sai schema hoặc sai entity.
  • premature_final: model trả lời trước khi gọi tool cần thiết.
  • policy_gap: prompt nói không làm, nhưng harness không enforce.
  • recovery_gap: tool/model lỗi nhưng loop retry hoặc fallback sai.
  • eval_gap: case test không bắt được lỗi đã xảy ra trong production.
