LLM AI Agents Tool Use

LLM Agents thực chiến — từ prompt đến autonomous tool use

TA Tuan Anh · 29 tháng 3, 2026 · ⏱ 12 phút đọc

Nếu Prompt Engineering giúp bạn lấy được câu trả lời tốt từ LLM trong một lượt gọi, thì LLM Agents đưa điều đó lên một tầm cao mới: model có thể suy luận nhiều bước, gọi tools bên ngoài, và tự quyết định hành động tiếp theo. Đây là nền tảng của nhiều hệ thống AI hiện đại, từ coding assistant đến autonomous research agents.

Agent khác gì với một prompt thông thường?

Một cuộc gọi LLM thông thường là stateless và one-shot: input → output. Ngược lại, agent chạy trong một loop: quan sát → suy nghĩ → hành động → quan sát kết quả → lặp lại cho đến khi đạt mục tiêu.

Các thành phần của một agent điển hình:

  • LLM: bộ não đưa ra quyết định.
  • Tools: các hàm bên ngoài agent có thể gọi (search, calculator, API, database).
  • Memory: lưu lại những gì đã xảy ra trong phiên làm việc.
  • Planning / Reasoning: chiến lược để phân rã task phức tạp.

ReAct — Reasoning + Acting

ReAct là pattern phổ biến nhất cho agent. Thay vì yêu cầu LLM trả lời ngay, bạn bắt model output theo chu kỳ Thought → Action → Observation. 

Question: Tổng dân số của Việt Nam và Thái Lan cộng lại là bao nhiêu?

Thought: Tôi cần tìm dân số của hai quốc gia này. Tôi sẽ dùng search tool.
Action: search("population of Vietnam 2024")
Observation: Dân số Việt Nam năm 2024 ước tính khoảng 100.3 triệu người.

Thought: Bây giờ tôi cần tìm dân số Thái Lan.
Action: search("population of Thailand 2024")
Observation: Dân số Thái Lan năm 2024 ước tính khoảng 71.8 triệu người.

Thought: Tôi có thể cộng hai con số: 100.3 + 71.8 = 172.1.
Action: calculate("100.3 + 71.8")
Observation: 172.1

Final Answer: Tổng dân số khoảng 172.1 triệu người.

Điểm mạnh của ReAct là interpretability: bạn có thể đọc được quá trình suy nghĩ của model, từ đó debug dễ hơn nhiều so với một câu trả lời bí ẩn.

Tool use / Function calling

Các model hiện đại như GPT-4, Claude, Gemini đều hỗ trợ function calling: model không trực tiếp chạy hàm, mà trả về JSON mô tả hàm cần gọi và arguments. Code của bạn sẽ thực thi hàm đó và đưa kết quả trở lại model. 

import json
from openai import OpenAI

client = OpenAI()

def search(query: str) -> str:
    # Giả lập search
    return f"Kết quả tìm kiếm cho: {query}"

tools = [{
    "type": "function",
    "function": {
        "name": "search",
        "description": "Tìm kiếm thông tin trên internet",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {"type": "string"}
            },
            "required": ["query"]
        }
    }
}]

messages = [
    {"role": "user", "content": "Tìm kiếm thông tin về vector database Qdrant."}
]

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    tools=tools,
)

tool_call = response.choices[0].message.tool_calls[0]
args = json.loads(tool_call.function.arguments)
result = search(args["query"])

messages.append(response.choices[0].message)
messages.append({
    "role": "tool",
    "tool_call_id": tool_call.id,
    "content": result,
})

final = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    tools=tools,
)
print(final.choices[0].message.content)

Việc tách biệt "quyết định gọi tool" và "thực thi tool" giúp bạn kiểm soát hoàn toàn những gì agent có thể làm — một yếu tố bảo mật rất quan trọng.

Xây dựng agent đơn giản từ đầu

Bạn không cần framework nặng để bắt đầu. Một agent loop tối thiểu trông như thế này: 

def run_agent(question: str, max_steps: int = 5) -> str:
    messages = [
        {"role": "system", "content": "Bạn là một assistant hữu ích. "
         "Suy nghĩ từng bước và gọi tool khi cần."},
        {"role": "user", "content": question}
    ]

    for _ in range(max_steps):
        response = client.chat.completions.create(
            model="gpt-4o-mini", messages=messages, tools=tools
        )
        msg = response.choices[0].message

        if not msg.tool_calls:
            return msg.content

        messages.append(msg)
        for tc in msg.tool_calls:
            fn_name = tc.function.name
            args = json.loads(tc.function.arguments)
            result = call_tool(fn_name, args)  # dispatcher của bạn
            messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "content": str(result),
            })

    return "Agent đã dừng sau số bước tối đa."

Memory trong agent

Agent không thông minh nếu không nhớ. Có hai loại memory chính:

  • Short-term memory: chính là conversation history đưa vào context window. Giới hạn bởi số token model hỗ trợ.
  • Long-term memory: lưu trữ ngoài, thường dùng vector database để retrieve thông tin liên quan khi cần. Đây là cách agent "học" từ các phiên trước.

Trong production, đừng chỉ append toàn bộ history vào context. Hãy dùng summarization hoặc retrieval để giữ context window hiệu quả.

Khi nào nên dùng agent?

Agent rất mạnh nhưng cũng dễ over-engineer. Mình dùng agent khi:

  • Task không thể hoàn thành trong một lượt gọi LLM.
  • Cần kết hợp nhiều nguồn dữ liệu hoặc tools khác nhau.
  • Cần model tự quyết định workflow dựa trên kết quả trung gian.

Và không dùng agent khi:

  • Một prompt đơn giản đã đủ.
  • Cần determinism cao (agent có thể chọn sai tool hoặc loop vô hạn).
  • Latency và cost quan trọng hơn flexibility.

Kết luận

LLM Agents mở ra một cách tiếp cận mới để xây dựng phần mềm: thay vì lập trình từng bước cụ thể, bạn định nghĩa mục tiêu, cung cấp tools, và để model tự quyết định. Tuy nhiên, đi kèm với sức mạnh là sự phức tạp: bạn cần theo dõi cost, latency, error handling và đặc biệt là bảo mật khi gọi external tools.
