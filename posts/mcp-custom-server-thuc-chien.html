MCP thực chiến — tự tạo custom server cho AI agent

Tuan Anh · 6 tháng 7, 2026 · ⏱ 13 phút đọc

Khi làm AI agent, phần khó không chỉ là prompt. Phần khó hơn là làm sao để model truy cập đúng dữ liệu, gọi đúng hành động, và vẫn giữ được quyền kiểm soát ở phía hệ thống. Model Context Protocol (MCP) sinh ra để giải quyết đúng khoảng trống đó: một chuẩn chung để AI application kết nối với file, database, API nội bộ và workflow của bạn.

Nếu ví function calling là cách bạn đưa vài hàm vào một model cụ thể, thì MCP là lớp giao thức để đóng gói các hàm, tài nguyên và prompt đó thành một server có thể dùng lại qua nhiều client: Claude Desktop, Claude Code, VS Code, Cursor hoặc client tự viết.

MCP là gì?

MCP là một chuẩn mở cho phép AI application kết nối tới hệ thống bên ngoài. Theo tài liệu chính thức, MCP cho phép ứng dụng AI truy cập data sources, tools và workflow thông qua một giao thức thống nhất. Cách hiểu thực dụng nhất: MCP là API adapter dành cho AI agent.

Thay vì mỗi app AI tự tích hợp riêng với GitHub, PostgreSQL, Figma, Jira hoặc hệ thống nội bộ, bạn viết một MCP server expose capability. Client nào hiểu MCP đều có thể discover và gọi capability đó theo cùng một protocol.

Kiến trúc: host, client, server

MCP dùng mô hình client-server nhưng có ba vai trò cần phân biệt:

  • MCP Host: ứng dụng AI người dùng đang dùng, ví dụ Claude Desktop hoặc một IDE có agent.
  • MCP Client: thành phần bên trong host, giữ kết nối tới từng MCP server.
  • MCP Server: process cung cấp context hoặc hành động cho host.

Một host có thể kết nối nhiều server cùng lúc. Ví dụ một coding assistant có thể dùng server filesystem để đọc repo, server database để inspect schema, và server Jira nội bộ để đọc ticket. Mỗi server chỉ cần tập trung vào một domain rõ ràng.

Data layer và transport layer

Về mặt giao thức, MCP tách thành hai lớp. Data layer dùng JSON-RPC 2.0 để mô tả lifecycle, capability negotiation, discovery và execution. Transport layer quyết định message đi qua đâu: local process qua stdio, hoặc remote server qua Streamable HTTP.

Với custom server chạy local, stdio là lựa chọn phổ biến nhất. Client launch process server, rồi trao đổi JSON-RPC qua stdin/stdout. Vì vậy có một rule rất quan trọng: server stdio không được log ra stdout. Log ra stdout sẽ làm hỏng stream JSON-RPC. Hãy dùng stderr hoặc file log.

Ba primitive chính: tools, resources, prompts

MCP server có thể expose ba loại capability chính:

  • Tools: hàm có side effect hoặc tính toán chủ động, ví dụ query database, tạo issue, gọi API, chạy phân tích.
  • Resources: dữ liệu dạng đọc, ví dụ file content, schema, record, log, report. Resource giống một URL mà client có thể đọc để thêm context.
  • Prompts: template tái sử dụng để hướng dẫn model làm một workflow cụ thể, ví dụ review PR, phân tích incident, viết release note.

Trong thực tế, hãy bắt đầu bằng tools. Khi tool ổn định, bổ sung resource để model có context tốt hơn, rồi thêm prompt khi bạn thấy người dùng lặp đi lặp lại cùng một workflow.

Khi nào nên tạo custom MCP?

Custom MCP server đáng làm khi bạn có ít nhất một trong các nhu cầu sau:

  • Agent cần truy cập dữ liệu nội bộ không nên public ra internet.
  • Nhiều AI client cần dùng cùng một integration.
  • Bạn muốn kiểm soát permission, logging, audit và validation ở một lớp riêng.
  • Workflow có logic domain riêng, không thể giải quyết chỉ bằng prompt.

Ngược lại, nếu use case chỉ là gọi một API duy nhất trong app của bạn, function calling trực tiếp có thể đơn giản hơn. MCP phát huy giá trị khi integration cần tái sử dụng, cần chạy cục bộ, hoặc cần tách biệt khỏi code của AI host.

Ví dụ: tạo MCP server đọc tri thức nội bộ

Ta sẽ tạo một server nhỏ tên team-knowledge. Server này expose hai tool: search_notes để tìm ghi chú theo keyword và get_note để đọc chi tiết một ghi chú. Ví dụ dùng TypeScript vì SDK chính thức khá gọn và hợp với các team web/backend.

1. Setup project

mkdir team-knowledge-mcp
cd team-knowledge-mcp
npm init -y
npm install @modelcontextprotocol/sdk zod@3
npm install -D typescript @types/node

Thêm cấu hình tối thiểu vào package.json:

{
  "type": "module",
  "bin": {
    "team-knowledge-mcp": "./build/index.js"
  },
  "scripts": {
    "build": "tsc && chmod 755 build/index.js"
  }
}

Và tsconfig.json:

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "outDir": "./build",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  },
  "include": ["src/**/*"]
}

2. Viết server

Tạo file src/index.ts:

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

type Note = {
  id: string;
  title: string;
  tags: string[];
  body: string;
};

const notes: Note[] = [
  {
    id: "deploy-checklist",
    title: "Production deploy checklist",
    tags: ["devops", "release"],
    body: "Run tests, verify migrations, check feature flags, monitor logs after deploy."
  },
  {
    id: "incident-template",
    title: "Incident review template",
    tags: ["sre", "incident"],
    body: "Impact, timeline, root cause, detection gap, action items, owners."
  }
];

const server = new McpServer({
  name: "team-knowledge",
  version: "1.0.0"
});

server.registerTool(
  "search_notes",
  {
    description: "Search internal team notes by keyword or tag.",
    inputSchema: {
      query: z.string().min(2).describe("Keyword or tag to search")
    }
  },
  async ({ query }) => {
    const q = query.toLowerCase();
    const matches = notes.filter((note) =>
      note.title.toLowerCase().includes(q) ||
      note.body.toLowerCase().includes(q) ||
      note.tags.some((tag) => tag.includes(q))
    );

    return {
      content: [{
        type: "text",
        text: matches.length
          ? matches.map((n) => `${n.id}: ${n.title} [${n.tags.join(", ")}]`).join("\n")
          : "No matching notes found."
      }]
    };
  }
);

server.registerTool(
  "get_note",
  {
    description: "Read one internal note by id.",
    inputSchema: {
      id: z.string().describe("Note id returned by search_notes")
    }
  },
  async ({ id }) => {
    const note = notes.find((item) => item.id === id);

    return {
      content: [{
        type: "text",
        text: note
          ? `# ${note.title}\nTags: ${note.tags.join(", ")}\n\n${note.body}`
          : `Note not found: ${id}`
      }]
    };
  }
);

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("team-knowledge MCP server running on stdio");
}

main().catch((error) => {
  console.error("Fatal error:", error);
  process.exit(1);
});

Điểm đáng chú ý là schema input dùng zod. Schema này không chỉ validate tham số ở runtime, mà còn giúp client hiểu tool cần gì để model gọi đúng hơn.

3. Build và test

npm run build
npx @modelcontextprotocol/inspector node ./build/index.js

MCP Inspector giúp bạn xem server expose tool nào, gọi thử input, đọc response và debug lỗi protocol mà không cần mở hẳn một AI client.

4. Cấu hình client

Với client hỗ trợ local MCP server, cấu hình thường có dạng command + args. Ví dụ cho một host dùng format mcpServers: 

{
  "mcpServers": {
    "team-knowledge": {
      "command": "node",
      "args": ["/ABSOLUTE/PATH/team-knowledge-mcp/build/index.js"]
    }
  }
}

Luôn dùng absolute path. Sau khi cấu hình, restart host để nó launch server và chạy bước capability discovery. Từ đó model có thể thấy search_notes và get_note như tools có thể gọi.

Thiết kế tool tốt cho LLM

Tool cho LLM không giống API cho con người. Model cần tên, mô tả và schema thật rõ. Một vài nguyên tắc mình thấy hiệu quả:

  • Tên tool cụ thể: dùng search_notes thay vì search.
  • Description nói rõ khi nào dùng, không chỉ nói tool làm gì.
  • Input nhỏ và typed: tránh nhận blob JSON quá lớn nếu không cần.
  • Output dễ đọc: trả text có cấu trúc, id ổn định, lỗi rõ ràng.
  • Idempotent nếu có thể: tool mutate dữ liệu nên yêu cầu xác nhận ở tầng client hoặc app.

Khi nào thêm resources và prompts?

Sau khi có tools, bạn có thể expose resource cho dữ liệu đọc thường xuyên. Ví dụ team://notes/deploy-checklist đọc checklist deploy, hoặc db://schema/public đọc schema database. Resource hữu ích khi context nên được đọc trực tiếp thay vì thông qua tool search.

Prompts thì phù hợp với workflow lặp lại. Ví dụ prompt incident_review có thể yêu cầu model đọc note template, hỏi thêm missing fields, rồi viết postmortem. Khi prompt sống trong MCP server, team có thể version và chia sẻ workflow thay vì copy paste prompt trong từng client.

Bảo mật: đừng coi MCP là magic sandbox

MCP giúp chuẩn hóa giao tiếp, nhưng không tự giải quyết toàn bộ bảo mật. Nếu server của bạn có quyền đọc file, query database hoặc gọi API production, agent cũng có thể đề xuất dùng những quyền đó. Cần thiết kế theo nguyên tắc least privilege.

  • Chạy server với credential scope hẹp nhất có thể.
  • Validate input bằng schema và kiểm tra lại ở code, đặc biệt với path, SQL, URL.
  • Không expose tool destructive nếu chưa có cơ chế xác nhận rõ.
  • Ghi audit log cho tool quan trọng, nhưng với stdio hãy log ra stderr hoặc file.
  • Với HTTP transport, dùng auth token/OAuth, HTTPS và kiểm soát redirect/SSRF.
  • Không đặt secret trong description của tool, resource hoặc prompt.

Một rủi ro thực tế là tool poisoning: description hoặc dữ liệu trả về có thể chứa instruction lừa model bỏ qua rule ban đầu. Vì vậy dữ liệu từ MCP server vẫn nên được coi là untrusted input nếu nó đến từ hệ thống bên ngoài hoặc người dùng khác.

Debug và đưa vào production

Checklist ngắn trước khi dùng MCP trong workflow thật:

  • Test từng tool bằng MCP Inspector.
  • Đảm bảo build artifact chạy được bằng command giống hệt config client.
  • Thêm timeout cho API/database call.
  • Trả lỗi có cấu trúc, không throw stack trace chứa secret.
  • Version server và ghi changelog khi đổi schema tool.
  • Theo dõi latency, số lần gọi tool và tỉ lệ lỗi.

Kết luận

MCP là một bước tiến quan trọng cho hệ sinh thái AI agent: nó biến integration thành một server chuẩn hóa, discoverable và tái sử dụng được. Với custom MCP server, bạn có thể đưa tri thức nội bộ, API riêng và workflow domain vào AI client mà không phải viết lại integration cho từng app.

Cách bắt đầu tốt nhất là chọn một use case nhỏ: search tài liệu team, đọc schema database, tạo release note, hoặc truy vấn hệ thống monitoring. Viết 1-2 tool thật rõ, test bằng Inspector, rồi mới mở rộng sang resources, prompts và remote transport.
