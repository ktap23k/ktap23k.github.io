Apache Airflow thực chiến — orchestrate data pipeline như một pro

TA Tuan Anh · 22 tháng 3, 2026 · ⏱ 13 phút đọc

Trong thế giới data-driven, hầu hết công việc của data engineer không phải là viết Spark job phức tạp, mà là đảm bảo hàng chục bước xử lý dữ liệu chạy đúng lịch, đúng thứ tự và có thể retry khi lỗi. Apache Airflow đã trở thành công cụ chuẩn để giải quyết bài toán đó — và sau vài năm dùng, mình nhận ra Airflow mạnh ở chỗ nó ép bạn nghĩ rõ ràng về workflow hơn là chỉ chạy script.

Airflow là gì và khi nào cần?

Airflow là một platform open-source để lập lịch và monitor workflows. Workflow trong Airflow được định nghĩa bằng code Python — gọi là DAG (Directed Acyclic Graph). Mỗi node trong DAG là một task, và các edge thể hiện dependency.

Bạn nên dùng Airflow khi:

  • Có nhiều bước xử lý dữ liệu phụ thuộc nhau (ETL/ELT pipeline).
  • Cần lập lịch chạy định kỳ (hàng giờ, hàng ngày, hàng tuần).
  • Cần retry, alerting và observability cho data jobs.
  • Muốn pipeline có thể tái sử dụng và version control.

Nếu chỉ có một vài script Python chạy bằng cron, Airflow có thể là overkill. Nhưng khi cron bắt đầu có 20 dòng và bạn không biết cái nào chạy trước cái nào, đó là lúc nên migrate.

Các thành phần cơ bản

DAG là container cho toàn bộ workflow. Task là một đơn vị công việc. Operator là template để tạo task — PythonOperator, BashOperator, PostgresOperator, v.v. Sensor là một loại operator đặc biệt chờ điều kiện nào đó thỏa mãn (file xuất hiện, query trả về kết quả...).

Kiến trúc Airflow gồm:

  • Scheduler: quét DAGs và quyết định task nào cần chạy.
  • Worker: thực thi task (có thể là LocalExecutor, Celery, Kubernetes).
  • Webserver: UI để theo dõi và trigger DAG.
  • Metadata database: PostgreSQL hoặc MySQL lưu trạng thái DAG runs.

DAG đầu tiên với TaskFlow API

Mình khuyến khích dùng TaskFlow API (có từ Airflow 2.0+) thay vì operator truyền thống vì code gọn, dễ đọc và tự động handle XCom. 

from airflow.decorators import dag, task
from airflow.utils.dates import days_ago
from datetime import timedelta

@dag(
    dag_id="etl_daily_orders",
    start_date=days_ago(1),
    schedule_interval="@daily",
    catchup=False,
    default_args={"retries": 2, "retry_delay": timedelta(minutes=5)},
    tags=["etl", "sales"],
)
def etl_daily_orders():

    @task
    def extract(date: str) -> list[dict]:
        # Đọc dữ liệu từ source
        return [{"id": 1, "amount": 100, "date": date}]

    @task
    def transform(rows: list[dict]) -> list[dict]:
        return [{**r, "amount_vat": r["amount"] * 1.1} for r in rows]

    @task
    def load(rows: list[dict]) -> None:
        print(f"Loaded {len(rows)} rows")

    load(transform(extract("{{ ds }}")))

etl_daily_orders()

Biến {{ ds }} là một Jinja template trả về logical date của DAG run. TaskFlow API tự động push/pull dữ liệu qua XCom, bạn không cần viết xcom_pull thủ công.

Scheduling, catchup và backfill

Đây là phần nhiều người nhầm lẫn nhất. start_date là ngày bắt đầu của DAG, schedule_interval là khoảng cách giữa các lần chạy. Airflow chạy DAG cho một interval đã kết thúc, không phải chạy ngay tại start_date.

  • Catchup: nếu catchup=True và bạn deploy DAG có start_date từ 1 năm trước, Airflow sẽ tạo run cho tất cả các interval chưa chạy. Trong production thường để catchup=False.
  • Backfill: chạy lại dữ liệu quá khứ bằng lệnh airflow dags backfill. Hữu ích khi bạn thay đổi logic và muốn tính lại dữ liệu cũ.

Idempotency — nguyên tắc vàng

Một task được gọi là idempotent nếu chạy nhiều lần vẫn cho cùng kết quả. Điều này cực kỳ quan trọng vì Airflow retry task khi lỗi. Nếu task thứ hai insert thêm dữ liệu thay vì upsert, bạn sẽ có duplicate.

Các pattern idempotent phổ biến:

  • Load dữ liệu theo partition: DELETE WHERE date = '{{ ds }}' rồi insert.
  • Dùng MERGE hoặc INSERT ... ON CONFLICT với PostgreSQL.
  • Đặt tên file output có chứa logical date để tránh ghi đè nhầm.

Sensors và external triggers

Thay vì chạy DAG theo lịch cố định, đôi khi bạn muốn chờ một file đến từ upstream hoặc một job bên ngoài hoàn thành. Sensors giúp làm điều đó: 

from airflow.providers.http.sensors.http import HttpSensor
from airflow.providers.filesystem.sensors.filesystem import FileSensor

wait_for_file = FileSensor(
    task_id="wait_for_source_file",
    filepath="/data/raw/orders_{{ ds }}.csv",
    fs_conn_id="fs_default",
    poke_interval=60,
    timeout=3600,
)

Sensors có thể chạy ở chế độ poke (liên tục kiểm tra, chiếm worker slot) hoặc reschedule (nhả slot khi đang chờ, tiết kiệm tài nguyên). Trên production, hãy dùng mode="reschedule".

Chạy Airflow trong production

Những điều mình luôn làm khi deploy Airflow:

  • Executor: LocalExecutor cho dev/test, CeleryExecutor hoặc KubernetesExecutor cho production. KubernetesExecutor giúp isolate resource theo task.
  • Secrets: đừng hardcode password trong DAG. Dùng Airflow Connections và Variables, hoặc tích hợp AWS Secrets Manager / Vault.
  • Resource: đặt pool và priority_weight để kiểm soát concurrency với các hệ thống bên ngoài.
  • SLA và alerting: cấu hình sla trên task và gửi alert qua Slack/Email khi DAG chậm hoặc fail.

Những sai lầm phổ biến

  • Đặt top-level code nặng trong file DAG: Airflow scheduler parse DAG liên tục. Nếu file DAG kết nối database ở top level, scheduler sẽ spam DB. Chuyển logic vào trong task function.
  • Pass object lớn qua XCom: XCom không phải data warehouse. Với dữ liệu lớn, lưu vào S3/MinIO và truyền URL qua XCom.
  • Dynamic tasks quá nhiều: expand() tạo hàng nghìn task map sẽ làm metadata DB phình to. Cân nhắc batch processing thay vì task per item.

Kết luận

Airflow không phải silver bullet, nhưng nếu bạn cần orchestrate workflows phức tạp với khả năng observability tốt, đây vẫn là lựa chọn an toàn. Hãy bắt đầu với TaskFlow API, giữ task idempotent, và đừng quên set catchup=False trước khi deploy lên production.
